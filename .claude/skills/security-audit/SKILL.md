# 安全审计

从多个方面检查项目代码的安全隐患，给用户一份说人话的审计报告。用户是编程小白，报告不要堆黑话，要说清楚"有什么风险、会怎样、怎么改"。

## 检查范围

如果用户指定了文件/目录就查它;没指定就查整个项目的源代码和配置文件（`electron/`、`src/`、根目录的配置文件等），跳过 `node_modules`、`dist`、`release`。

## 四个检查方向

### 方向一:敏感信息泄露(硬编码密钥)

在代码里搜有没有直接写死的敏感信息:
- 密码、API key、token、密钥、证书私钥;
- 数据库连接串里的账号密码;
- 第三方服务的 secret / access key（如云服务、支付、短信）;
- 形如 `password = "xxx"`、`apiKey: "..."`、`token = "..."` 的硬编码。

搜索关键词参考:`password`、`passwd`、`secret`、`api_key`/`apikey`、`token`、`private_key`、`access_key`、`credential` 等（大小写都查）。找到就指明文件和行号,说明为什么危险。

### 方向二:注入类漏洞(SQL 注入等)

重点查数据库操作和命令执行:
- **SQL 注入**:SQL 语句是不是用字符串拼接把变量塞进去的（如 `"... WHERE id=" + id` 或模板串 `` `...${x}...` ``)。正确做法是用参数化查询(占位符 `?` + 参数数组)。本项目用 sql.js,要确认都走 `db.run(sql, [参数])` 这种带占位符的写法,而不是把用户输入拼进 SQL 文本。
- **命令注入**:执行系统命令(`spawn`/`exec`)时有没有把外部输入直接拼进命令字符串。
- **路径穿越**:用外部传入的文件名拼路径读写文件时,有没有校验(防 `../../` 跑到别处)。

### 方向三:配置文件里的明文敏感信息

检查配置类文件:
- `package.json`、各种 `*.config.js`、`.env`、`*.json`、`*.yml` 等;
- 有没有明文写着密码、密钥、私有仓库凭证、签名证书密码等;
- `.env` 这类含密钥的文件有没有被 `.gitignore` 排除掉（不该提交进 Git)。
- 检查 `.gitignore` 是否覆盖了敏感文件和数据库文件。

### 方向四:其他安全隐患(结合本项目)

本项目是 Electron + React + sql.js 桌面记账软件，重点看:
- **Electron 安全配置**:`main.js` 里创建窗口时,`nodeIntegration` 是否关闭、`contextIsolation` 是否开启、有没有用 preload 做隔离。这是 Electron 最常见的安全点。
- **preload / IPC**:`preload.js` 暴露给界面的接口是不是最小必要,有没有把危险能力（如任意执行、任意读写文件）直接开放给前端。
- **导入的数据是否校验**:导入 Excel/JSON 时,外部文件内容有没有做基本校验(类型、数值合法性),防止脏数据或恶意内容。
- **依赖风险**:`package.json` 里有没有明显可疑、疑似仿冒(typosquatting)或年久失修的依赖。可提示用户用 `npm audit` 查已知漏洞。
- **数据文件安全**:本地数据库 xiaobai.db 是否明文存储敏感信息(记账软件一般无高敏数据,如实说明即可)。

## 输出:审计报告

给用户一份说人话的报告,包含:

1. **总体结论**:整体安全状况如何（有没有高危问题、大体放心还是需要处理）。
2. **按严重程度列问题**,每条写清:
   - 风险等级:🔴高危 / 🟡中等 / 🔵提示;
   - 在哪个文件第几行(用 `文件:行号` 格式,可点击);
   - 属于哪个方向(敏感信息 / 注入 / 配置明文 / 其他);
   - 具体是什么风险、会导致什么后果(用大白话讲,比如"别人拿到这个能干嘛");
   - 建议怎么改。
3. **没问题的地方也说一句**:哪些方面检查过、是安全的,让用户放心。

排序:高危(能直接被利用、泄露密钥、注入)排最前,提示类排后。

## 注意事项

- 默认**只审计、只报告,不自动改代码**。除非用户明确说"帮我修",才动手。
- 读到疑似密钥/密码的内容,报告里用"第几行有一处疑似密钥"指代,**不要把密钥原文完整抄进报告**,避免二次泄露。
- 不确定是不是真漏洞时,如实说明"这里有风险点,需确认",不要夸大也不要漏报。
- 这是防御性的安全检查,目的是帮用户加固自己的软件。
