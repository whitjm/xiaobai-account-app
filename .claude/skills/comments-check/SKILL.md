# 注释检查

检查项目代码的注释质量,给用户一份说人话的检查报告。用户是编程小白,报告不要堆黑话。

## 检查范围

1. 如果用户指定了某个文件/目录,就查它;没指定就查项目里主要的源代码文件:
   - 前端:`src/` 下的 `.tsx`/`.ts` 文件
   - Electron 后端:`electron/` 下的 `.ts` 文件
   - 跳过 `node_modules`、`dist`、`src/types/`、测试文件等。
2. 逐个文件读代码,按下面三条标准检查。

## 三条检查标准

### 标准一:注释数量是否足够

要求:**每个函数、每条重要的核心代码都要有详细注释**。参考密度是"10 行代码里大约 3 行注释、7 行正文"(约 30%),作为大致标尺,不必逐行死抠。

检查时:
- 统计每个文件的注释行数占比(注释行 ÷ 代码总行数,空行不算)。TypeScript 注释以 `//` 或 `/*...*/` 计。
- 重点看**函数**:每个函数上方或内部有没有说明它是干嘛的。没有注释或只有一行说明的函数要点名。
- 重点看**核心逻辑**:IPC 通信、数据库 CRUD、状态管理、事件处理等有没有解释。
- 明显偏低(远低于 30%)或关键函数裸奔(完全没注释)的,列为问题。

### 标准二:注释和代码是否匹配

检查注释说的和代码实际干的是不是一回事:
- 注释描述的行为与代码实际逻辑不符(代码改了、注释没跟着改)。
- 注释里的变量名、函数名、类名和代码对不上。
- 过时注释:还停留在旧版本、描述的是已经删掉或改掉的逻辑。
- 复制粘贴留下的错误注释(说的是别处的事)。

这类"对不上"的注释最坑人,要重点揪出来,指明第几行、错在哪。

### 标准三:注释是否符合小白视角

本项目面向编程小白,注释应该让不太懂技术的人也能大致看懂:
- 好注释:解释"这段在干嘛、为什么这么做、输入输出是什么",用大白话。
- 差注释:只是把代码翻译一遍(如 `i++ // i 加一`),等于没说;或满是黑话、缩写、专业术语而不解释。
- 如果只有一句 `/** Foo. */` 没有实质内容,也算不够。

## 输出:检查报告

读完代码后,给用户一份说人话的报告,包含:

1. **总体情况**:查了哪几个文件、整体注释水平怎么样(好/一般/偏少)。
2. **按文件列问题**,每条写清楚:
   - 在哪个文件第几行(用 `文件:行号` 格式,可点击);
   - 属于三条标准里的哪一条(数量不足 / 对不上 / 小白看不懂);
   - 具体问题是什么;
   - 建议怎么改(可以给一句改后的注释示范)。
3. **注释数量小结**:各文件的注释占比大概是多少,哪些达标、哪些偏低。
4. 问题按严重程度排序:注释和代码对不上(会误导人)最严重,其次是关键函数没注释,最后是可优化的表述。

## 注意事项

- 默认**只检查、只报告,不自动改代码**。除非用户明确说"顺便帮我补上/改好",才动手改注释。
- 只碰注释,任何情况下都不改动代码逻辑本身。
- 如果代码注释整体不错,如实说"写得挺好",不要为了凑问题硬找茬。
