# 单元测试:创建 + 执行 + 报告

给代码里的核心逻辑（一个个函数/零件）写自动化检查程序，跑一遍，核对结果对不对，最后给用户一份成绩单。

用户是编程小白：解释要用大白话（把"函数"说成"零件"，把"断言"说成"核对答案"），报告要说人话。

## 前提:测试工具

本项目是 **Electron 桌面应用 + React 前端 + SQLite 数据库**的记账软件。

- **前端 React/TypeScript 逻辑**：统一用 **Vitest**（项目已配置）。测试文件放在 `src/utils/__tests__/` 目录，命名 `*.test.ts`。
- **Electron 后端 TypeScript 逻辑**：同样用 Vitest，测试文件放在 `electron/__tests__/` 目录。

第一次运行本技能时，确认工具已安装：
- `npm install -D vitest`（项目已有）
- 运行测试：`npm run test`
- 覆盖率报告：`npm run test:coverage`

## 操作步骤

### 第一步:确定要测什么

1. 如果用户指定了某个文件/功能，就测它；没指定就先看项目结构，挑**纯逻辑、好测**的部分优先。本项目首选：
   - `src/utils/dateRange.ts`（日期范围计算：本周/本月/本年/自定义）
   - `src/utils/validation.ts`（金额校验、分类名校验）
   - `src/utils/categoryIcons.ts`（分类图标映射）
   - `electron/validators.ts`（输入校验、字符串清理）
   - `electron/queries.ts`（数据库 CRUD 逻辑）
2. 用大白话告诉用户这次打算测哪几个"零件"、每个零件负责啥。
3. 避开难测的部分（依赖真实数据库文件、文件系统、Electron IPC 的代码）——必要时用"假数据/替身"隔离，或先跳过并说明原因。**不要在测试里真的去连接 SQLite 数据库文件。**

### 第二步:创建单元测试

1. 前端测试文件统一放在 `src/utils/__tests__/` 目录，命名形如 `test_xxx.ts`。
2. 每个"零件"写几条检查，覆盖：正常情况、边界情况（空数据、0、负数）、异常情况。
3. 用 Vitest 写法：`describe` 分组，`test` 或 `it` 定义用例，`expect(实际).toBe(期望)`；同类用例可用 `test.each` 批量。
4. 测试用中文注释/docstring 写清楚"在测什么"。
5. 若被测模块 import 时会触发副作用（如初始化数据库），直接从具体模块文件导入被测函数，避免连带加载。

### 第三步:执行测试

1. 运行测试：`npm run test`
2. 耗时通常几秒。
3. 如需看详细输出：`npm run test -- --reporter=verbose`

### 第四步:给出测试报告

跑完后，用大白话给用户一份成绩单，包含：
- 一共测了多少项，通过多少、失败多少；
- 如果全过：明确说"全部通过，这几个零件干得没问题"；
- 如果有失败：逐条说清哪个零件、测的是什么情况、期望的结果 vs 实际结果，并判断是**代码有 bug** 还是**测试写得不对**，给出修复建议；
- 附上关键的原始输出（失败的那几条），方便核对。

## 注意事项

- 测试是"只读验证"，不改动被测的业务代码；除非确认是代码 bug 且用户同意，才去改业务代码。
- 测试用的临时文件，跑完清理干净。
- 不要在单测里连真实的 SQLite 数据库文件；用假数据或替身隔离，或明确跳过并说明原因。
- 不确定某段代码该测什么时，先问用户这个零件"本该有什么行为"，再照着写检查。
