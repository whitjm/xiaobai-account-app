# 单元测试:创建 + 执行 + 报告

给代码里的核心逻辑（一个个函数/零件）写自动化检查程序，跑一遍，核对结果对不对，最后给用户一份成绩单。

用户是编程小白：解释要用大白话（把"函数"说成"零件"，把"断言"说成"核对答案"），报告要说人话。

## 前提:测试工具

本项目基于 Vite + React + Electron。单元测试统一用 **Vitest**（和 Vite 同源，配置最省心）。

第一次运行本技能时，先检查是否已安装：
1. 看 `package.json` 的 devDependencies 里有没有 `vitest`。
2. 若没有，安装：`npm install -D vitest`。需要测 React 组件时再加 `@testing-library/react @testing-library/jest-dom jsdom`。
3. 在 `package.json` 的 scripts 里加上（若还没有）：`"test": "vitest run"`（跑一次出结果）和 `"test:watch": "vitest"`（改代码自动重跑）。
4. 如依赖下载失败，用国内镜像重试。

## 操作步骤

### 第一步:确定要测什么

1. 如果用户指定了某个文件/功能，就测它；没指定就先看项目结构，挑**纯逻辑、好测**的部分优先，例如 `electron/queries.js` 里的合计计算、筛选、增删改查等纯函数。
2. 用大白话告诉用户这次打算测哪几个"零件"、每个零件负责啥。
3. 避开难测的部分（依赖真实数据库、文件系统、Electron 窗口的代码）——必要时用"假数据/替身"隔离，或先跳过并说明原因。

### 第二步:创建单元测试

1. 测试文件放在被测文件旁边或统一的 `tests/` 目录，命名形如 `xxx.test.js`。
2. 每个"零件"写几条检查，覆盖：正常情况、边界情况（如空数据、0、负数）、异常情况。
3. 用 Vitest 的写法：`describe` 分组、`it`/`test` 一条检查、`expect(...).toBe(...)` 核对答案。测试描述用中文写清楚"在测什么"。

### 第三步:执行测试

1. 运行 `npm test`（即 `vitest run`），跑完就结束、输出结果。
2. 耗时通常几秒到几十秒。

### 第四步:给出测试报告

跑完后，用大白话给用户一份成绩单，包含：
- 一共测了多少项，通过多少、失败多少；
- 如果全过：明确说"全部通过，这几个零件干得没问题"；
- 如果有失败：逐条说清哪个零件、测的是什么情况、期望的结果 vs 实际结果，并判断是**代码有 bug** 还是**测试写得不对**，给出修复建议。
- 附上关键的原始输出（失败的那几条），方便核对。

## 注意事项

- 测试是"只读验证"，不改动被测的业务代码；除非确认是代码 bug 且用户同意，才去改业务代码。
- 测试用的临时文件、假数据库文件，跑完清理干净。
- 不确定某段代码该测什么时，先问用户这个零件"本该有什么行为"，再照着写检查。
